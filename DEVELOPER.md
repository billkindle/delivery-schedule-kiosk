## Architecture Overview

```
┌─────────────────────────────────────────────────────────┐
│  Browser (Edge kiosk / any browser)                     │
│                                                         │
│  React SPA ──── polls /api/events every 5 min ─────▶   │
│                                                         │
└────────────────────────┬────────────────────────────────┘
                         │ HTTP (same origin)
┌────────────────────────▼────────────────────────────────┐
│  Express server  (server/index.js)                      │
│                                                         │
│  GET /api/events                                        │
│    1. Check in-memory token cache                       │
│    2. If expired → POST /oauth2/v2.0/token → Entra ID   │
│    3. GET /users/{upn}/calendarView → Microsoft Graph   │
│    4. Return JSON array of events                       │
│                                                         │
│  Static file serving (dist/) in production              │
└─────────────────────────────────────────────────────────┘
```

**Key design decisions:**
- The Express server is a thin proxy — it only fetches, never mutates.
- OAuth token is cached in memory with a 60-second expiry buffer. No Redis or persistent cache needed.
- The React app uses polling (not WebSockets) because the underlying calendar data changes infrequently and simplicity is a priority for kiosk deployment.
- The `/calendarView` Graph endpoint is used instead of `/events` so recurring events are automatically expanded into individual instances within the date window.

## Development Setup

### Requirements

- Node.js 18+
- A `.env` file with valid credentials, **or** use demo mode (see [Demo Mode](#demo-mode))

### Install and run

```bash
npm install
npm run dev
```

This starts two processes concurrently:
- **Vite dev server** on `http://localhost:5173` — React app with HMR
- **Express API server** on `http://localhost:3001` — handles `/api/*`

Vite proxies all `/api` requests to `localhost:3001` (configured in `vite.config.js`).

```
Browser → http://localhost:5173
  └── /api/events → proxied to → http://localhost:3001/api/events
```

In production (after `npm run build`), Express serves both the built React app and the API from a single port (`3000` by default).

### Available scripts

| Command | Description |
|---|---|
| `npm run dev` | Start Vite + Express for local development |
| `npm run build` | Build React app into `dist/` |
| `npm start` | Start Express server (serves `dist/` + API) |
| `npm run dev:client` | Start only the Vite dev server |
| `npm run dev:server` | Start only the Express API server |

## Backend — Express Server

### `server/index.js`

Responsibilities:
- Loads `.env` via `dotenv/config`
- Serves the built React app from `dist/` (only if that directory exists — skips in dev)
- Exposes `GET /api/events` — returns `{ events: [...], fetchedAt: "ISO string" }`
- Exposes `GET /api/health` — returns `{ ok: true, ts: "ISO string" }`
- Provides `GET /api/events?demo=1` — returns hardcoded mock events (no credentials needed)
- Falls back all non-API routes to `dist/index.html` (SPA routing support)

### `server/graphClient.js`

Responsibilities:
- Manages the OAuth 2.0 client credentials token lifecycle
- Token is cached in memory with a 60-second expiry buffer
- Fetches calendar events via the `/calendarView` endpoint

**Token flow:**

```
getCalendarEvents()
  └── getAccessToken()
        ├── if (cache valid) → return cached token
        └── else → POST https://login.microsoftonline.com/{tenant}/oauth2/v2.0/token
                    body: client_id, client_secret, scope, grant_type=client_credentials
                  ← { access_token, expires_in }
                  → store in tokenCache, return token
```

**Graph API call:**

```
GET https://graph.microsoft.com/v1.0/users/{MAILBOX_UPN}/calendarView
  ?startDateTime={today 00:00 UTC}
  &endDateTime={today + LOOKAHEAD_DAYS UTC}
  &$select=id,subject,start,end,location,bodyPreview,categories,organizer,isAllDay
  &$orderby=start/dateTime
  &$top=150
Headers:
  Authorization: Bearer {token}
  Prefer: outlook.timezone="UTC"
```

The `Prefer: outlook.timezone="UTC"` header ensures all `start.dateTime` and `end.dateTime` values come back in UTC, so the frontend can format them consistently using the browser's local timezone.

**Why `/calendarView` and not `/events`?**
The `/events` endpoint returns the recurring event series definition (e.g., "every Tuesday"). The `/calendarView` endpoint expands recurring events into individual instances within the requested window — essential for a calendar display.

## Frontend — React App

### State management

The app uses React's built-in `useState` and `useEffect`. There is no external state library — the data needs are simple enough that prop drilling through two levels is cleaner than introducing Redux or Zustand.

**State in `App.jsx`:**

```js
const [events, setEvents]       // Array of Graph event objects
const [loading, setLoading]     // Boolean — true until first fetch completes
const [error, setError]         // String | null — last error message
const [lastFetch, setLastFetch] // Date | null — timestamp of last successful fetch
```

### Refresh mechanism

```js
const REFRESH_MS = 5 * 60 * 1000  // 5 minutes

useEffect(() => {
  fetchEvents()                          // Fetch immediately on mount
  const id = setInterval(fetchEvents, REFRESH_MS)
  return () => clearInterval(id)         // Clean up on unmount
}, [fetchEvents])
```

### Rendering pipeline

```
App.jsx
  ├── fetches /api/events every 5 minutes
  ├── Header.jsx     (receives: lastFetch, hasError)
  ├── DeliveryBoard.jsx  (receives: events, loading, error)
  │     ├── groups events by local calendar date
  │     ├── TodayScroller  (auto-scrolls every 15s)
  │     │     └── EventCard × N  (large variant)
  │     └── Upcoming rows
  │           └── EventCard × N  (compact variant)
  └── <footer>  (logo)
```

### `<EventCard event palette large>`

| Prop | Type | Description |
|---|---|---|
| `event` | `Object` | Graph calendar event object |
| `palette` | `Object` | Color theme `{ border, bg, text, badge }` |
| `large` | `boolean` | `true` = today card (288px wide), `false` = upcoming compact card (224px wide) |

**`getPalette(event, fallbackIndex)`** (exported from EventCard.jsx)

Returns a palette object. Priority:
1. If `event.categories[0]` exists → hash the string to a palette index (consistent color per category name)
2. Otherwise → use `fallbackIndex % 8`

This means if you tag Outlook calendar events with "Route 1", "Route 2", etc., they always get the same color across all days.

## Data Flow

### Raw Graph event → display

A Graph `/calendarView` response looks like:

```json
{
  "id": "AAMkAGI...",
  "subject": "Riverside Chemical – Monthly Replenishment",
  "isAllDay": false,
  "start": { "dateTime": "2026-06-08T07:30:00.0000000", "timeZone": "UTC" },
  "end":   { "dateTime": "2026-06-08T09:00:00.0000000", "timeZone": "UTC" },
  "location": { "displayName": "1420 Riverside Industrial Pkwy, Springfield" },
  "bodyPreview": "Full pallet. Forklift required...",
  "categories": ["Route 2"],
  "organizer": { "emailAddress": { "name": "Scheduler", "address": "scheduler@fabrikam.com" } }
}
```

**Transform pipeline in `DeliveryBoard.jsx`:**

1. `groupByDate(events)` — converts `start.dateTime` UTC string to a local `YYYY-MM-DD` key using `Intl`
2. `assignPalettes(groups)` — maps each event to a `{ ev, palette }` pair with a globally incrementing index (so colors don't reset between days)
3. `todayGroup` / `upcomingGroups` — split by whether the date key matches today

**Time display in `EventCard.jsx`:**

```js
new Date(dateStr).toLocaleTimeString('en-US', {
  hour: 'numeric', minute: '2-digit', hour12: true, timeZone: 'UTC',
})
```

The `timeZone: 'UTC'` here is intentional — Graph returns datetimes already in UTC (due to the `Prefer` header), so we format them directly rather than converting. This means the times shown match what was entered in Outlook.

> **If times appear 1 hour off:** The event creator likely used a different timezone in Outlook and the `Prefer` header override isn't being respected. Check Graph Explorer to confirm the raw `start.dateTime` value.

## Demo Mode

Demo mode bypasses Microsoft Graph entirely and returns hardcoded mock events from the server. Useful for:
- UI development without credentials
- Showing stakeholders without a live mailbox
- Testing layout with a specific number of events

**Activate:** Append `?demo=1` to the URL.

**How it works:**

```
Browser → GET /api/events?demo=1
  └── server checks req.query.demo === '1'
        └── returns mockEvents() instead of calling getCalendarEvents()
```

The `mockEvents()` function in `server/index.js` generates events relative to `new Date()` so they always appear as today and upcoming regardless of when the server is started.

To add/modify mock events, edit the `mockEvents()` function in `server/index.js`.

### Azure App Service

**First-time setup:**

- Set `NODE_ENV=production` and all required env vars in App Settings (or Key Vault references)
- Set the startup command to install dependencies and start the server:

  ```text
  npm install --production && node server/index.js
  ```

  Azure does **not** run `npm install` automatically — the startup command handles it.
- The platform injects `PORT` automatically; the server reads it via `process.env.PORT`

**Deploy (first-time and every update):**

```powershell
# 1. Build the React app
npm run build

# 2. Create deployment ZIP — no node_modules, no .env
Compress-Archive -Path `
  .\dist, .\server, .\package.json, .\package-lock.json `
  -DestinationPath deploy.zip -Force

# 3. Deploy to Azure
az webapp deployment source config-zip `
  --name <app-name> `
  --resource-group <resource-group> `
  --src deploy.zip
```

> **Note:** Use `az webapp deployment source config-zip` (not `az webapp deploy`).
> The newer `az webapp deploy` command returns intermittent 400 errors with this
> project's ZIP structure. The older command uses the `/api/zipdeploy` endpoint
> and is reliable. Do not set `SCM_DO_BUILD_DURING_DEPLOYMENT=true` — it causes
> Oryx to attempt a redundant build that interferes with deployment.

The first cold start after deploy is slower (~30s) because `npm install --production`
runs before the server starts. Subsequent requests are fast.

## Known Limitations & Gotchas

### Timezone handling

The server uses `Prefer: outlook.timezone="UTC"` so all event times come back in UTC. The frontend formats them using `timeZone: 'UTC'` in `Intl.DateTimeFormat`. This means:

- Times displayed match what was entered in Outlook (Exchange converts to UTC on the server side)
- The kiosk machine's Windows timezone setting does NOT affect how event times are displayed
- All-day events use date-only strings (`2026-06-08T00:00:00`) with no timezone context — they display as "All Day"

If users report times being off by exactly N hours, the likely cause is that the event organizer saved events without specifying a timezone in Outlook. Check the raw `start.timeZone` field via Graph Explorer.

### No real-time push

Events are polled every 5 minutes (`REFRESH_MS` in `App.jsx`). If a last-minute delivery is added to the calendar, it may take up to 5 minutes to appear on the board. This is intentional — Microsoft Graph change notifications (webhooks) require a publicly reachable HTTPS endpoint and add significant complexity for marginal benefit on a display screen.

To change the interval:
```js
// src/App.jsx
const REFRESH_MS = 5 * 60 * 1000  // Change to desired interval in ms
```

### Token stored in memory only

The OAuth token cache is in-process memory. If the Node server restarts (e.g., after a deploy or crash), the next API call will fetch a new token from Entra. This adds ~200–500ms to the first request after a restart. It is not a correctness issue.

### Graph API rate limits

Microsoft Graph throttles at approximately 10,000 requests per 10 minutes per application. At a 5-minute poll interval with a single kiosk, you'll make ~288 requests/day — well within limits. If you scale to many kiosks, consider adding server-side caching with a shared TTL.

### `dist/` must be built before starting the server

Running `npm start` without first running `npm run build` will start the server with no frontend. The server detects this gracefully (skips static serving) but the kiosk will show nothing. Always run `npm run build` after deploying new code.

### Edge kiosk mode and localhost

Windows Assigned Access with Edge in digital signage mode starts Edge before all system services are fully ready. If the Node service hasn't started yet, Edge will show a "page can't be reached" error. The recommended fix is configuring a startup delay (see README.md Step A.4). Edge will automatically reload once the server is up if you configure the Edge kiosk settings to refresh on error.

## Common Development Tasks

### Add a new field to event cards

1. Add the field to the `$select` parameter in `server/graphClient.js`:
   ```js
   $select: 'id,subject,start,end,location,bodyPreview,categories,organizer,isAllDay,myNewField'
   ```
2. Access it in `EventCard.jsx` via `event.myNewField`
3. Add it to the mock events in `server/index.js` for demo mode

### Change the auto-scroll interval

In `src/components/DeliveryBoard.jsx`:
```js
const SCROLL_INTERVAL_MS = 15_000  // Change to desired interval in ms
```

### Change the data refresh interval

In `src/App.jsx`:
```js
const REFRESH_MS = 5 * 60 * 1000  // Change to desired interval in ms
```

### Add a new brand color token

1. Add it to the `cs` object in `tailwind.config.js`:
   ```js
   cs: {
     teal: '#007B6E',  // hypothetical new token
   }
   ```
2. Use it in components: `bg-cs-teal`, `text-cs-teal`, `border-cs-teal`
3. If the class is assembled dynamically, add it to the `safelist` in `tailwind.config.js`

### Change the number of days shown

Either update the `.env` file:
```
LOOKAHEAD_DAYS=7
```

Or change the default in `server/graphClient.js`:
```js
const { ..., LOOKAHEAD_DAYS = '7' } = process.env
```

### Add a different calendar (multiple mailboxes)

In `server/graphClient.js`, fetch from multiple mailboxes and merge the results:

```js
export async function getCalendarEvents() {
  const token = await getAccessToken()
  const mailboxes = process.env.MAILBOX_UPN.split(',').map(s => s.trim())
  const results = await Promise.all(mailboxes.map(upn => fetchEventsForMailbox(token, upn, params)))
  return results.flat().sort((a, b) => a.start.dateTime.localeCompare(b.start.dateTime))
}
```

Set `MAILBOX_UPN=mailbox1@fabrikam.com,mailbox2@fabrikam.com` in `.env`.

## Dependency Notes

### Why no React Query / SWR?

The data fetching needs are minimal — one endpoint, one component consuming it, simple polling. Adding a data-fetching library would be over-engineering for the current scope.

### Why Express instead of a serverless function?

The app needs to serve both the API and the static React files from a single process for simplicity on a Windows kiosk (one `npm start` command, one Windows service). Serverless would require a separate static hosting layer. If moving to Azure, Azure Functions + Azure Static Web Apps is a valid future architecture.

### Why Vite instead of Create React App?

CRA is deprecated. Vite is faster, actively maintained, and produces smaller bundles due to better tree-shaking and native ES modules.

### Updating dependencies

```bash
npm outdated        # See what's stale
npm update          # Update within semver ranges
npx npm-check-updates -u && npm install   # Update to latest (review changelog first)
```

Key packages to watch:
- `vite` — check for breaking changes in major versions
- `tailwindcss` — v4 has significant config changes from v3
- `express` — v5 is in RC; currently on v4
- `dotenv` — stable, rarely changes
